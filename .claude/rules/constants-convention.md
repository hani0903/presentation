# 상수(Constants) 추출 판단 기준

요구사항에 등장하는 값을 상수로 추출할지, 컴포넌트 내부에 인라인으로 둘지를 판단하는 기준이다.

## 기본 원칙

"관리 포인트인가?" 하나로 판단한다. 관리 포인트가 아니면 상수로 추출하지 않는다.

## 관리 포인트로 판정 (→ 상수로 추출)

아래 중 **하나 이상** 해당하면 상수로 추출한다.

### 1. 요구사항이 명시적으로 지시

요구사항 본문에 아래 키워드가 포함된 값은 **반드시** 상수로 추출한다:

- `"별도로 설정값으로 관리"`
- `"관리 포인트"` / `"관리포인트"`
- `"상수로 관리"` / `"상수로 정의"`
- `"설정으로 분리"` / `"설정값으로 분리"`

이는 **패턴 매칭 룰**이다. 위 키워드가 등장한 값은 해석 여지 없이 상수로 뽑는다.

### 2. 비즈니스·도메인 의미가 있는 값

- 비즈니스 제약: `MAX_POST_LENGTH`, `IDLE_TIMEOUT_MS`, `MAX_MANNER_KEYWORDS`
- 사용자 노출 문구: `placeholder`, 에러 메시지, 버튼 레이블
- 도메인 식별자: `DEFAULT_TIER`, query key 첫 세그먼트, API endpoint path prefix

### 3. 여러 곳에서 참조

두 개 이상의 파일·컴포넌트에서 동일한 값을 사용하면 상수로 추출한다. 단, 단순 레이아웃 수치는 예외 (각 컴포넌트마다 맥락이 다를 수 있음).

### 4. 외부에서 오버라이드 가능해야 하는 값

- prop 기본값으로 노출되는 placeholder
- 환경·브랜드별로 다른 값 (`import.meta.env.PUBLIC_*`)

## 관리 포인트가 아님 (→ 인라인)

아래 값은 **구체적 수치가 명시되어 있어도** 상수로 뽑지 않고 컴포넌트에 인라인한다.

### 1. 레이아웃·스타일 수치

padding, margin, gap, 아이콘 크기, border-radius, font-size 등. 요구사항에 `"여백은 16px이다"`, `"간격은 8px이다"` 같은 문구는 **디자인 스펙**으로 해석한다.

- Tailwind 클래스로 표현 가능한 값은 Tailwind 우선 (`px-4`, `gap-2`, `py-3`)
- Tailwind로 표현 어려운 값은 인라인 `style` 또는 arbitrary value (`gap-[6px]`)

### 2. 단일 컴포넌트 내에서만 쓰이는 값

다른 곳에서 재사용되지 않는 값은 상수로 뽑지 않는다.

### 3. 순수 구현 디테일

- HTML input `rows={1}`, textarea line height 등 구현에 종속된 값
- 내부 애니메이션 duration (요구사항에 명시 없는 경우)

## 판단 플로우

```
요구사항 값을 보면:
  ├─ 키워드 ("설정값으로 관리" 등) 있음? → 상수 ✅
  ├─ 비즈니스 의미 있음 (제약, 사용자 문구)? → 상수 ✅
  ├─ 여러 곳에서 쓰임 (2+ 위치)? → 상수 ✅
  ├─ 레이아웃·스타일 수치? → 인라인 ❌
  └─ 그 외 단일 컴포넌트 내부 값? → 인라인 ❌
```

## 예시

### ✅ 상수로 추출

```ts
// 요구사항: "placeholder 값은 별도로 설정값으로 관리한다"
export const POST_INPUT_PLACEHOLDER = "어떤 내용을 작성하시겠어요?";

// 요구사항: "최대 게시글 길이 500자" — 비즈니스 제약
export const MAX_POST_LENGTH = 500;

// 여러 컴포넌트에서 참조하는 값
export const PROFILE_CARD_HEIGHT = 120;
```

### ❌ 상수로 뽑지 않음 (인라인 유지)

```tsx
// 요구사항: "상단의 좌우 여백은 16px이고 상하 여백은 12px이다" — 레이아웃 스펙
<div className="px-4 py-3" />

// 요구사항: "간격은 6px"
<div className="gap-[6px]" />

// ❌ Bad — 레이아웃 수치를 불필요하게 추출
export const TOP_PADDING_X = 16;
export const TOP_PADDING_Y = 12;
```

## 의심스러울 때

"상수로 뺄까?"가 고민되면 **인라인이 기본**이다. 나중에 재사용이 생기거나 관리 포인트로 승격되면 그때 추출한다. 반대로 상수로 먼저 만든 뒤 인라인으로 되돌리는 것은 더 번거롭고, 요구사항 변경 이력을 오염시킬 가능성이 높다.
