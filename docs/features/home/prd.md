# 홈 페이지 PRD

> 참조 스펙: `docs/features/home/spec-fixed.md`
> 작성일: 2026-05-30

---

## 개요

### 목적

개인 발표 자료 보관소(Presentation Archive)의 진입점을 제공한다. 방문자가 페이지에 진입했을 때 사이트의 성격을 즉시 이해하고, 준비된 발표 자료로 이동할 수 있도록 한다.

### 배경

Claude Code 하네스를 활용한 컴포넌트 자동화 파이프라인을 직접 시연하기 위해 제작한 발표 보조 앱이다. 발표 현장에서 실제로 동작하는 앱을 보여줌으로써, 하네스 기반 개발 워크플로우의 실용성을 증명하는 것이 목적이다.

### 범위

- 대상 화면: `/` (루트 경로), `/harness` (발표 슬라이드 placeholder)
- 구현 형태: 정적 UI (서버 연동 없음)
- 우선 환경: 데스크톱 발표 환경

### 성공 기준

- 페이지 진입 시 "Presentation Archive"임을 즉시 인식할 수 있다.
- 발표 카드를 클릭하면 `/harness`가 새 탭으로 열린다.
- 다크모드가 기본으로 적용되며 토글로 전환 가능하다.
- 라이트/다크 모드 양쪽에서 시각적 오류 없이 표시된다.

## 사용자 스토리

### US-01 사이트 진입

> **As a** 발표를 보러 온 청중 또는 발표자 본인  
> **I want to** 사이트 이름과 소개 문구를 보고 싶다  
> **So that** 이 사이트가 무엇인지 즉시 파악할 수 있다

**수용 기준**

- 사이트 이름 `Presentation Archive`가 눈에 띄는 크기로 표시된다.
- 사이트를 한 줄로 소개하는 문구가 제목 아래에 표시된다.
- 콘텐츠 전체가 화면 수직 중앙에 정렬된다.

---

### US-02 발표 카드 확인

> **As a** 발표 자료를 찾는 방문자  
> **I want to** 어떤 발표가 준비되어 있는지 한눈에 보고 싶다  
> **So that** 원하는 발표 자료로 바로 이동할 수 있다

**수용 기준**

- 발표 카드가 화면에 표시된다.
- 카드에는 제목, 한 줄 설명, 키워드 배지가 포함된다.
- 카드에 마우스를 올리면 시각적 피드백(배경색 변화)이 제공된다.

---

### US-03 발표 자료로 이동

> **As a** 발표를 진행하는 발표자  
> **I want to** 발표 카드를 클릭해 슬라이드 화면으로 이동하고 싶다  
> **So that** 홈 화면을 유지한 채 새 탭에서 발표 자료를 열 수 있다

**수용 기준**

- 발표 카드 클릭 시 `/harness`가 **새 탭**으로 열린다.
- 홈 페이지는 그대로 유지된다.
- `/harness` 경로에는 placeholder 페이지가 존재한다.

---

### US-04 다크모드 전환

> **As a** 발표 환경에 따라 화면 밝기를 조정하고 싶은 발표자  
> **I want to** 라이트/다크 모드를 즉시 전환하고 싶다  
> **So that** 빔 프로젝터나 조명 환경에 맞게 화면을 최적화할 수 있다

**수용 기준**

- 우상단에 다크모드 토글 버튼이 표시된다.
- 기본값은 다크 모드다 (localStorage 저장값 없을 경우).
- 토글 클릭 시 라이트/다크 모드가 전환된다.
- 선택한 테마가 localStorage에 저장되어 새로고침 후에도 유지된다.

## 기술 결정 (ADR)

> 채택 구현 안: **안 B(FSD Entity 분리) + 안 C(Custom Hook 추상화) 혼합**
>
> - 발표 자료를 `entities/presentation` 슬라이스로 분리하여 FSD 레이어 원칙을 준수한다.
> - 데이터 접근은 `usePresentations()` 커스텀 훅으로 추상화하여 API 전환 비용을 최소화한다.

---

### ADR-01 발표 카드 데이터 관리 방식

| 항목             | 내용                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Context**      | 발표 카드 목록을 어느 레이어에, 어떤 형태로 배치할지 결정해야 한다. 현재는 정적 상수 배열이지만 향후 API 연동 가능성이 있다. FSD 아키텍처에서 데이터가 위치하는 레이어가 의존 방향을 결정하므로, 잘못 배치하면 나중에 대규모 이동이 발생한다.                                                                                                                                                                                          |
| **Decision**     | `entities/presentation` 슬라이스를 신설하고, 타입(`model/types.ts`)·상수(`model/constants.ts`)·커스텀 훅(`model/usePresentations.ts`)·표시 컴포넌트(`ui/PresentationCard.tsx`)를 배치한다. `pages/home/HomePage`는 `usePresentations()`만 호출하고 데이터 소스를 직접 알지 못한다.                                                                                                                                                     |
| **Alternatives** | ① **`pages/home` 자기 완결형(안 A)** — 거부 이유: FSD에서 pages는 조합 레이어이므로 비즈니스 데이터를 두면 레이어 책임 위반. 다른 페이지에서 재사용 불가. ② **`shared/config`에 전역 상수** — 거부 이유: `shared`는 도메인 무관 코드만 위치해야 한다는 FSD 원칙 위반. 발표 자료는 도메인 데이터다. ③ **훅 없이 상수를 HomePage에서 직접 import** — 거부 이유: API 전환 시 컴포넌트를 수정해야 하고, 테스트에서 상수 모킹이 복잡해진다. |
| **Consequences** | ✅ FSD 레이어 원칙 준수. API 전환 시 `usePresentations` 내부만 교체하면 되며 컴포넌트 수정 범위가 없다. `PresentationCard`는 props 기반으로 독립 단위 테스트 가능. ⚠️ 현재 정적 카드 1개를 위해 슬라이스를 신설하므로 초기 ceremony가 과다해 보일 수 있다. 발표 카드가 영구적으로 정적 데이터로 남는다면, entities 레이어의 본래 설계 의도(도메인 CRUD 엔티티)와 맞지 않는다는 비판을 받을 수 있다.                                    |

### ADR-02 다크모드 기본값 처리 방식

| 항목             | 내용                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ---------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Context**      | spec에서 "localStorage에 저장값이 없을 경우 dark로 초기화"를 요구한다. 현재 `useTheme`의 `getInitialTheme()`은 localStorage → 시스템 설정(`prefers-color-scheme`) 순으로 fallback한다. 발표자의 시스템이 라이트 모드이면 첫 진입 시 라이트로 렌더되어 요구사항을 위반한다.                                                                                                                                                                                                                |
| **Decision**     | `getInitialTheme()`의 최종 fallback을 `"dark"`로 고정한다. localStorage에 저장된 값이 있으면 그 값을 존중하고, 없으면 시스템 설정과 무관하게 항상 `"dark"`를 반환한다.                                                                                                                                                                                                                                                                                                                    |
| **Alternatives** | ① **시스템 설정 fallback 유지(현재 구현)** — 거부 이유: 발표자 시스템이 라이트 모드이면 spec 위반. 발표 직전 모드를 확인해야 하는 운영 부담 발생. ② **`<html class="dark">`를 `index.html`에 하드코딩** — 거부 이유: localStorage에 저장된 라이트 모드 선택이 새로고침 후 무시됨. 사용자 선택을 파괴하는 UX 안티패턴. ③ **CSS `prefers-color-scheme: dark`만으로 처리** — 거부 이유: "항상 dark 기본값" 요구를 CSS만으로 구현 불가. 사용자 토글 기능과 함께 쓰려면 JS 상태 관리가 필수다. |
| **Consequences** | ✅ 발표 환경에서 항상 일관된 다크 모드로 시작. localStorage를 통해 사용자 선택이 새로고침 후에도 유지됨. ⚠️ 시스템이 라이트 모드인 방문자는 첫 진입 시 시스템 설정과 다른 화면을 보게 된다. 발표 보조 앱이라는 특수 목적에는 적합하지만, 일반 서비스 기준으로는 시스템 설정 무시가 UX 안티패턴으로 분류된다.                                                                                                                                                                              |

### ADR-03 `/harness` 새 탭 라우팅

| 항목             | 내용                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Context**      | 발표 카드 클릭 시 `/harness`로 이동해야 한다. 발표 중 홈 화면을 유지한 채 슬라이드를 별도 탭에서 여는 것이 발표 흐름에 유리하다. React Router의 `<Link>`는 SPA 내부 라우팅(같은 탭 이동)이므로 요구사항과 충돌한다.                                                                                                                                                                                                                    |
| **Decision**     | `<a href="/harness" target="_blank" rel="noreferrer">`를 사용한다. 내비게이션 의도이지만 새 탭 제어가 필요하므로 `<Link>` 대신 앵커 태그를 명시적으로 사용한다.                                                                                                                                                                                                                                                                        |
| **Alternatives** | ① **React Router `<Link to="/harness">`** — 거부 이유: 같은 탭에서 SPA 전환이 발생한다. 발표 중 홈으로 돌아오려면 브라우저 뒤로가기가 필요해 발표 흐름이 끊긴다. ② **`<Link>` + `useNavigate` + `window.open()`** — 거부 이유: `target="_blank"`로 충분한 것을 JS 이벤트 핸들러까지 추가하는 불필요한 복잡도. 팝업 차단 정책에도 더 취약하다.                                                                                          |
| **Consequences** | ✅ 발표 중 홈 탭과 슬라이드 탭을 독립적으로 유지 가능. 구현이 단순하고 의도가 명확하다. ⚠️ `rel="noreferrer"` 누락 시 Reverse Tabnapping 보안 취약점 발생 가능 — 반드시 함께 명시해야 한다. `target="_blank"`는 일부 브라우저 팝업 차단 설정에 따라 막힐 수 있으나, 로컬 발표 환경에서는 무관하다. React Router `<Link>`와 일반 `<a>`가 코드베이스에 혼재하므로, 내부 링크와 외부(새 탭) 링크를 구분하는 컨벤션을 팀 내 명시해야 한다. |

### ADR-04 `/harness` placeholder 페이지 구현 방식

| 항목             | 내용                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Context**      | 발표 카드가 `/harness`를 새 탭으로 열지만, 실제 발표 슬라이드 내용은 이번 구현 범위가 아니다. 라우트가 없으면 404가 발생하고, 발표 중 404 화면이 노출되면 오류 상황으로 오인될 수 있다.                                                                                                                                                                                                                                                                                                                                                                   |
| **Decision**     | `pages/harness` 슬라이스를 신설하고 최소 placeholder 페이지를 제공한다. "준비 중" 안내 메시지와 홈으로 돌아가는 링크만 포함한다. 실제 슬라이드 내용은 별도 구현 범위로 분리한다.                                                                                                                                                                                                                                                                                                                                                                          |
| **Alternatives** | ① **라우트 없이 404 허용** — 거부 이유: 발표 중 404 화면이 노출되면 의도된 상황인지 오류인지 청중이 구분 불가. 발표 신뢰도에 영향을 준다. ② **실제 슬라이드 내용을 이번 버전에 함께 구현** — 거부 이유: 이번 구현 범위(홈 페이지)를 벗어난다. 슬라이드 내용은 별도 REQ로 관리해야 범위가 명확해진다. ③ **React Router `errorElement`로 커스텀 404 처리** — 거부 이유: `errorElement`는 예상치 못한 라우팅 오류용이다. "준비 중인 페이지"를 표현하는 의미와 맞지 않으며, 정상 라우트로 등록된 페이지와 오류 처리를 혼용하는 것은 유지보수 혼란을 야기한다. |
| **Consequences** | ✅ 발표 중 안전한 fallback 제공. 향후 실제 슬라이드 내용으로 채워도 라우트 구조 변경 없이 컴포넌트만 교체하면 된다. ⚠️ 실제 콘텐츠 없는 페이지가 배포되므로, URL을 직접 접근하는 사용자에게는 빈 페이지로 보인다. placeholder임을 명시하는 메시지가 없으면 오류 페이지와 구분이 안 된다.                                                                                                                                                                                                                                                                  |

## Out of Scope

### 인증 / 계정

- 로그인 UI: 제공하지 않는다.
- 회원가입 UI: 제공하지 않는다.
- 소셜 로그인(Google, GitHub 등): 구현하지 않는다.
- 세션 유지 / 자동 로그인 처리: 구현하지 않는다.
- 비로그인·로그인 사용자 화면 분기: 구현하지 않는다.

### 서버 연동

- API 호출: 홈 페이지는 어떤 서버 요청도 하지 않는다.
- 발표 카드 목록의 동적 로딩: 서버에서 받아오지 않고 상수 배열로 관리한다.
- 사용자 데이터 표시(조회수, 좋아요, 북마크 등): 구현하지 않는다.

### 발표 카드 기능

- 검색: 카드 목록에서 발표 자료를 검색하는 기능은 제공하지 않는다.
- 필터링: 키워드·날짜·카테고리 기준 필터링은 제공하지 않는다.
- 카테고리 분류: 카테고리 탭 또는 그룹핑 기능은 제공하지 않는다.
- 정렬: 최신순·인기순 등 정렬 기능은 제공하지 않는다.
- 카드 상세 페이지: 카드 클릭 시 상세 페이지로 이동하는 방식은 사용하지 않는다(`/harness` 직접 이동).
- 카드 즐겨찾기 / 핀 고정: 구현하지 않는다.
- 카드 공유 링크 복사: 구현하지 않는다.

### `/harness` 페이지

- 실제 발표 슬라이드 내용: 이번 범위에서는 placeholder만 제공한다.
- 슬라이드 내비게이션(이전/다음): 구현하지 않는다.
- 발표자 노트 뷰: 구현하지 않는다.
- 전체화면 모드: 구현하지 않는다.

### 개인화 / 설정

- 언어 선택: 한국어 단일 언어만 제공한다.
- 폰트 크기 조절: 제공하지 않는다.
- 레이아웃 밀도(compact/comfortable) 설정: 제공하지 않는다.
- 알림 설정 진입점: 제공하지 않는다.

### 네비게이션

- 상단 GNB(Global Navigation Bar): 홈 페이지에는 별도 네비게이션 바를 두지 않는다.
- 브레드크럼(Breadcrumb): 제공하지 않는다.
- 푸터(Footer): 구현하지 않는다.
- 사이드바: 구현하지 않는다.

### 콘텐츠

- 서비스 소개 영상 / 데모 GIF: 제공하지 않는다.
- FAQ / 도움말 섹션: 제공하지 않는다.
- 업데이트 이력 / 릴리스 노트: 제공하지 않는다.
- SNS 공유 버튼: 제공하지 않는다.
- 발표 자료 미리보기 썸네일: 제공하지 않는다.

### 성능 최적화 (이번 버전)

- 이미지 에셋 최적화: 이번 버전에서는 이미지 에셋이 없으므로 해당 없음.
- 번들 최적화 / Code Splitting: 홈 페이지 단독 최적화는 하지 않으며 Vite 기본 빌드 설정을 따른다.
- PWA / 오프라인 지원: 구현하지 않는다.
- SEO 최적화(메타 태그, OG 태그): 구현하지 않는다.

### 접근성 (이번 버전)

- 스크린 리더 전용 최적화: 기본 시맨틱 마크업 수준만 제공한다.
- 키보드 단축키: 제공하지 않는다.
- 고대비 모드 대응: 구현하지 않는다.

## 용어 정의

| 용어                     | 정의                                                                                  |
| ------------------------ | ------------------------------------------------------------------------------------- |
| **Presentation Archive** | 이 사이트의 이름. 개인 발표 자료를 모아두는 보관소를 의미한다.                        |
| **발표 카드**            | 발표 자료 1건을 나타내는 카드 UI. 제목·설명·키워드 배지로 구성된다.                   |
| **키워드 배지**          | 발표 주제를 나타내는 작은 태그/칩 형태의 UI 요소.                                     |
| **placeholder 페이지**   | 실제 콘텐츠가 완성되기 전에 라우트 오류를 방지하기 위해 임시로 제공하는 최소 페이지.  |
| **하네스(Harness)**      | Claude Code의 실행 환경. 이 프로젝트에서는 `/harness` 경로의 발표 자료가 그 주제다.   |
| **다크모드 기본값**      | localStorage에 저장된 테마가 없을 때 적용되는 초기 테마. 이 앱에서는 dark로 고정한다. |
| **새 탭 이동**           | `target="_blank"`로 링크를 열어 현재 탭을 유지한 채 새 탭에서 목적지를 여는 방식.     |
| **정적 UI**              | 서버 API를 호출하지 않고 하드코딩된 데이터만으로 렌더링되는 화면.                     |
| **상수 배열**            | 코드 변경 없이는 수정되지 않는 고정 데이터 목록. 발표 카드 목록이 이에 해당한다.      |
