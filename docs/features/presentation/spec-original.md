# 프레젠테이션 페이지 기능 정의서

## 기능 개요

사용자가 홈 페이지의 발표 카드를 클릭했을 때, 해당 발표 자료를 볼 수 있는 프레젠테이션 페이지를 제공한다.

초기 버전에서는 `/harness` 경로에서 `Design to Commit` 발표를 보여준다.  
이번 구현 범위에서는 슬라이드 레이아웃과 네비게이션 구조만 구현하고, 각 슬라이드의 상세 콘텐츠는 추후 HTML 콘텐츠로 연결할 수 있도록 설계한다.

## 기능 요구사항

- 홈 페이지의 `Design to Commit` 카드를 클릭하면 `/harness` 경로로 이동한다.
- `/harness` 페이지는 발표용 슬라이드 화면으로 구성한다.
- 슬라이드는 16:9 발표 화면을 기준으로 설계한다.
- 사용자는 이전/다음 슬라이드로 이동할 수 있다.
- 키보드 방향키 `←`, `→`로 슬라이드를 이동할 수 있다.
- 화면 좌우에 이전/다음 이동 컨트롤을 제공한다.
- 현재 슬라이드 번호와 전체 슬라이드 수를 표시한다.
- 슬라이드 내부 콘텐츠는 추후 HTML 파일 또는 HTML 조각으로 교체 가능하도록 설계한다.
- 초기 버전에서는 실제 발표 내용 대신 placeholder 콘텐츠를 사용한다.

| ---------------- | -------------------------------------------------------------- |

## 슬라이드 구조

초기 버전에서는 다음과 같은 슬라이드 구성을 전제로 한다.

| 순서 | 슬라이드 역할       |
| ---- | ------------------- |
| 1    | 문제 제기           |
| 2    | 제안하는 파이프라인 |
| 3    | 각 기술 소개        |
| 4    | 현재 구현된 것      |
| 5    | 도전해보고 싶은 것  |
| 6    | 기대 효과           |
| 7    | 한 줄 요약          |

## UI / UX 요구사항

- 발표자는 데스크톱 화면에서 발표 자료를 명확하게 읽을 수 있어야 한다.
- 슬라이드는 화면 중앙에 배치한다.
- 슬라이드 콘텐츠는 발표 화면에서 충분히 큰 타이포그래피와 여백을 가져야 한다.
- 좌우 이동 컨트롤은 발표를 방해하지 않을 정도로 작게 배치한다.
- 키보드 네비게이션을 우선 지원한다.
- 다크 모드 기반 화면을 기본으로 한다.
- 디자인 시스템의 Presentation Typography / Spacing 기준을 따른다.
- 슬라이드 전환은 과하지 않은 기본 전환만 사용한다.
- `prefers-reduced-motion` 사용자를 고려한다.

## 네비게이션 요구사항

- `ArrowRight` 입력 시 다음 슬라이드로 이동한다.
- `ArrowLeft` 입력 시 이전 슬라이드로 이동한다.
- 첫 번째 슬라이드에서 이전 이동을 시도하면 아무 동작도 하지 않는다.
- 마지막 슬라이드에서 다음 이동을 시도하면 아무 동작도 하지 않는다.
- 마우스 클릭으로도 이전/다음 이동이 가능해야 한다.
- 현재 슬라이드 위치를 알 수 있도록 `현재 / 전체` 형식의 인디케이터를 제공한다.

## 콘텐츠 연결 방식

- 초기 버전에서는 슬라이드 데이터를 상수 배열로 관리한다.
- 각 슬라이드는 `id`, `title`, `description`, `contentHtml` 또는 `contentKey`를 가질 수 있다.
- 실제 발표 콘텐츠는 추후 HTML 파일 또는 HTML 조각으로 분리하여 연결할 수 있도록 한다.
- 이번 구현에서는 HTML 파일 로딩 기능을 필수로 구현하지 않는다.
- 슬라이드 렌더링 구조는 추후 콘텐츠 소스가 변경되어도 유지될 수 있어야 한다.

## 비고

- 초기 버전에서는 서버 연동을 제외한다.
- 발표 검색, 발표 목록 필터링, 발표 공유 기능은 구현하지 않는다.
- PDF Export, Speaker Note, 발표자 모드는 구현하지 않는다.
- 모바일 최적화는 우선순위에서 제외한다.
- 이번 버전의 핵심은 슬라이드 레이아웃과 네비게이션 구조 구현이다.
