# 홈 페이지 PRD

> 참조 스펙: `docs/features/home/spec-fixed.md`
> 작성일: 2026-05-29

---

## 개요

### 목적

서비스에 처음 진입하는 사용자가 Loopin의 목적을 즉시 이해하고, 퀴즈 풀이를 시작하거나 학습 자료에 접근할 수 있는 진입점을 제공한다.

### 배경

Loopin은 프론트엔드 개발자를 위한 학습 관리 및 퀴즈 풀이 앱이다. 홈 페이지는 사용자가 서비스를 처음 마주치는 화면으로, 서비스의 정체성을 전달하고 핵심 기능(퀴즈)으로 안내하는 역할을 한다.

### 범위

- 대상 화면: `/` (루트 경로)
- 구현 형태: 정적 UI 중심 (서버 연동 없음)
- 우선 플랫폼: 모바일 퍼스트

### 성공 기준

- 사용자가 페이지 진입 후 퀴즈 시작 버튼을 발견하고 `/quiz`로 이동할 수 있다.
- 외부 문서 카드를 클릭하면 새 탭으로 해당 문서가 열린다.
- 라이트/다크 모드 양쪽에서 시각적 오류 없이 표시된다.

## 사용자 스토리

### US-01 서비스 진입

> **As a** 처음 방문하는 사용자  
> **I want to** 서비스 이름과 소개 문구를 보고 싶다  
> **So that** 이 서비스가 무엇인지 즉시 파악할 수 있다

**수용 기준**

- 헤더에 서비스 이름 `Loopin`이 표시된다.
- 서비스를 한 줄로 소개하는 문구가 헤더 아래에 표시된다.
- 페이지 진입 시 콘텐츠가 페이드인 애니메이션으로 나타난다.

---

### US-02 퀴즈 시작

> **As a** 학습을 시작하려는 사용자  
> **I want to** 퀴즈를 바로 시작하고 싶다  
> **So that** 메뉴를 탐색하지 않고도 핵심 기능으로 진입할 수 있다

**수용 기준**

- 시작 버튼이 홈 페이지에 명확하게 표시된다.
- 시작 버튼 클릭 시 `/quiz` 경로로 이동한다.

---

### US-03 학습 자료 접근

> **As a** 학습 중인 사용자  
> **I want to** 공식 문서로 빠르게 이동하고 싶다  
> **So that** 별도로 검색하지 않고 참고 자료를 바로 열 수 있다

**수용 기준**

- 문서 링크 카드 목록이 표시된다.
- 각 카드는 문서 이름, 한 줄 설명, 외부 링크 아이콘(↗)을 포함한다.
- 카드 클릭 시 해당 URL이 새 탭(`target="_blank"`, `rel="noreferrer"`)으로 열린다.
- 초기 제공 카드는 4개(MDN, React, TypeScript, JavaScript 모던 튜토리얼)이다.

---

### US-04 다크 모드 대응

> **As a** 다크 모드를 사용하는 사용자  
> **I want to** 시스템 설정에 맞는 화면을 보고 싶다  
> **So that** 별도로 테마를 설정하지 않아도 눈에 편한 UI를 사용할 수 있다

**수용 기준**

- `prefers-color-scheme: dark` 환경에서 다크 테마가 자동 적용된다.
- `prefers-color-scheme: light` 환경에서 라이트 테마가 자동 적용된다.
- 두 모드 모두에서 텍스트 가독성과 대비가 유지된다.

## 기술 결정 (ADR)

> 채택 구현 안: **안 B — FSD 표준 분해**
> 채택 이유: 카드 단독 테스트 가능, FSD segment 경계 준수, API 전환 경계 명확

### ADR-01 라우팅 방식

| 항목             | 내용                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Context**      | 시작 버튼 클릭 시 `/quiz`로 이동해야 한다. 프로젝트는 React Router v7을 사용하며, 전체 페이지 새로고침 없이 SPA 내비게이션을 유지해야 한다.                                                                                                                                                                                                                                                                                                                          |
| **Decision**     | React Router v7의 `<Link to="/quiz">` 컴포넌트를 사용한다. 버튼처럼 보이더라도 내비게이션 의도를 가진 요소는 `<a>` 기반의 `<Link>`로 구현해 시맨틱과 접근성을 확보한다.                                                                                                                                                                                                                                                                                              |
| **Alternatives** | ① `<a href="/quiz">` — SPA 라우팅이 아닌 풀 페이지 리로드 발생, 불채택 ② `useNavigate()` + `onClick` — 단순 이동에 JS 핸들러가 불필요하게 개입, 불채택 ③ TanStack Router의 `<Link>` — 현재 프로젝트 라우터가 React Router v7이므로 혼용 불가, 불채택                                                                                                                                                                                                                 |
| **Consequences** | ✅ SPA 전환 유지 → 페이드인 애니메이션이 이탈 없이 동작. `<Link>`는 `<a>` 태그를 렌더하므로 키보드 내비게이션·스크린 리더 접근성 자동 확보. 향후 라우트 경로 변경 시 `to` prop 하나만 수정하면 된다. ⚠️ React Router 의존성이 생김 → 라우터를 교체할 경우 모든 `<Link>` 사용처를 일괄 수정해야 한다. 내부 링크(`<Link>`)와 외부 링크(`<a>`)가 코드에서 다른 컴포넌트로 분리되므로, 개발자가 실수로 외부 링크에 `<Link>`를 사용하지 않도록 팀 컨벤션 명시가 필요하다. |

### ADR-02 애니메이션 구현 방식

| 항목             | 내용                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ---------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Context**      | 페이지 진입 시 콘텐츠가 페이드인으로 나타나야 한다. 프로젝트는 Tailwind CSS v4를 사용하며, 런타임 JS 비용을 최소화하고 `prefers-reduced-motion` 사용자도 고려해야 한다.                                                                                                                                                                                                                                                                                                                                     |
| **Decision**     | Tailwind CSS의 커스텀 `animate-fade-in` 클래스를 `main` 태그에 적용한다. keyframe은 `@keyframes fadeIn { from { opacity: 0; transform: translateY(8px); } to { opacity: 1; transform: translateY(0); } }` 형태로 전역 CSS에 1회 정의한다. `prefers-reduced-motion: reduce` 미디어 쿼리로 애니메이션을 비활성화한다.                                                                                                                                                                                         |
| **Alternatives** | ① Framer Motion — 번들 크기(~50KB) 대비 단순 페이드인에 과도한 의존성, 불채택 ② JS 상태 기반(`useState` + `useEffect`) — CSS만으로 해결 가능한 것을 JS로 처리, 불필요한 리렌더 유발, 불채택 ③ `tw-animate-css` 라이브러리의 기존 클래스 — 프로젝트에 설치되어 있으나, 커스텀 easing·duration 요구 시 오버라이드 불편, 불채택                                                                                                                                                                                |
| **Consequences** | ✅ JS 없는 CSS 전용 구현 → 런타임 비용 0. `prefers-reduced-motion` 처리를 CSS 한 줄로 해결. 컴포넌트 테스트 시 애니메이션 상태를 별도로 모킹할 필요 없음. 향후 진입 방향(좌→우, 아래→위 등)을 keyframe 변경만으로 조정 가능. ⚠️ JS로 애니메이션을 제어(일시정지·중단·재실행)할 수 없다. 전역 CSS에 keyframe을 정의하므로 이름 충돌에 주의해야 한다(`fadeIn` 같은 범용 이름 사용 금지). 테스트에서 `prefers-reduced-motion` 미디어 쿼리를 모킹해야 접근성 분기를 검증할 수 있어 테스트 설정이 다소 번거롭다. |

### ADR-03 다크모드 구현 방식

| 항목             | 내용                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Context**      | `prefers-color-scheme` 시스템 설정에 따라 라이트/다크 모드를 자동 적용해야 한다. 이번 버전에서는 사용자가 직접 테마를 전환하는 UI를 제공하지 않는다. 프로젝트의 디자인 시스템은 CSS 변수 기반 토큰을 사용한다.                                                                                                                                                                                                                                                                   |
| **Decision**     | Tailwind CSS v4의 `dark:` variant(미디어 쿼리 방식)를 사용한다. 색상은 하드코딩 대신 디자인 시스템 토큰(`--color-surface-container`, `--color-on-surface` 등 CSS 변수)을 참조한다.                                                                                                                                                                                                                                                                                               |
| **Alternatives** | ① Tailwind `class` 전략 (`.dark` 클래스 토글) — JS로 클래스를 수동 관리해야 하므로 "시스템 설정 자동 추적"이라는 현재 요구사항에 불필요한 복잡도 추가, 불채택. 단, 향후 사용자 테마 전환 UI를 추가할 때는 `class` 전략으로 전환 검토 ② React Context로 테마 상태 관리 — 서버 연동 없는 정적 페이지에서 오버엔지니어링, 불채택 ③ 별도 다크모드 CSS 파일 분리 — 토큰 관리 포인트가 늘어나 디자인 시스템과 싱크 깨질 위험, 불채택                                                   |
| **Consequences** | ✅ JS 없는 CSS 미디어 쿼리 방식 → 시스템 테마 변경에 즉시 반응. 디자인 시스템 토큰을 참조하므로 토큰이 바뀌면 홈 페이지도 자동 반영. ⚠️ 사용자가 시스템 설정과 다른 테마를 선호해도 앱 내에서 직접 전환할 수 없다(이번 버전 Out of Scope이지만 UX 한계). 향후 사용자 토글 UI를 추가할 때 `class` 전략으로 전환하면 CSS 변수 주입 방식 변경이 추가로 필요하다. 테스트에서 다크모드를 검증하려면 `prefers-color-scheme` 미디어 쿼리를 모킹해야 하므로 테스트 환경 설정이 필요하다. |

### ADR-04 문서 링크 카드 데이터 관리

| 항목             | 내용                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Context**      | 4개의 문서 링크 데이터를 관리해야 한다. 스펙에서 "상수 배열로 관리"를 명시했으며, 현재는 서버 연동 없이 정적 데이터로 운영한다. 향후 서버에서 동적으로 목록을 받아올 가능성이 있다.                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| **Decision**     | `pages/home/model/constants.ts`에 `DocLink` 인터페이스와 `DOC_LINKS` 상수 배열을 정의한다. `DocLinkCard` 컴포넌트는 `DocLink` 타입의 props를 받아 렌더링하며, 데이터 출처를 알지 못한다. `HomePage`가 `DOC_LINKS`를 import해 카드 목록에 전달한다.                                                                                                                                                                                                                                                                                                                                                                                           |
| **Alternatives** | ① 컴포넌트 내 인라인 정의 — API 전환 시 컴포넌트 수술 필요, 카드 컴포넌트 재사용 불가, 불채택 ② `shared/config/`에 전역 상수로 배치 — 홈 페이지 전용 데이터를 공용 레이어에 두면 FSD 의존 방향 위반(`shared`는 도메인 무관 코드만), 불채택 ③ Zod 스키마 런타임 검증 — 컴파일타임 타입 검사로 충분한 정적 상수에 런타임 비용 추가, 과도한 복잡도, 불채택. 서버 응답 검증이 필요해지는 시점에 도입 검토                                                                                                                                                                                                                                        |
| **Consequences** | ✅ API 전환 경계 명확: 서버 연동 시 `HomePage`의 import를 `DOC_LINKS` 상수에서 `useDocLinksQuery()` 훅으로 교체하는 것만으로 충분하며 `DocLinkCard`는 변경 없음. FSD 준수: 데이터는 `model/`, 렌더링은 `ui/`로 segment 책임 분리. `DocLinkCard`는 props만으로 동작하므로 상수 배열 없이 단독 렌더 테스트 가능. ⚠️ 타입 검사는 컴파일타임에만 동작하므로 URL 오타나 깨진 링크는 런타임에 감지되지 않는다(사용자가 직접 클릭해야 발견). 새 카드를 추가하거나 삭제하려면 코드 수정 후 배포가 필요하다(서버 데이터였다면 배포 없이 관리 가능). 카테고리가 늘어나면 `constants.ts` 단일 파일이 비대해질 수 있어 분리 시점을 별도로 판단해야 한다. |

## Out of Scope

### 인증 / 계정

- 로그인 기능: 홈 페이지에서 로그인 UI를 제공하지 않는다.
- 회원가입 기능: 홈 페이지에서 회원가입 진입점을 제공하지 않는다.
- 소셜 로그인(Google, GitHub 등): 구현하지 않는다.
- 세션 유지 / 자동 로그인 처리: 구현하지 않는다.
- 비로그인 사용자와 로그인 사용자의 화면 분기: 구현하지 않는다.

### 서버 연동

- API 호출: 홈 페이지는 어떤 서버 요청도 하지 않는다.
- 사용자 데이터 표시(학습 진행률, 최근 풀이 기록 등): 구현하지 않는다.
- 퀴즈 목록 미리 표시: 홈에서 퀴즈 데이터를 fetch하지 않는다.
- 문서 카드 목록의 동적 로딩: 서버에서 받아오지 않고 상수 배열로 관리한다.

### 개인화 / 설정

- 사용자별 테마 설정 저장: 시스템 설정(`prefers-color-scheme`)만 따르고, 별도 설정 UI는 없다.
- 언어 선택: 한국어 단일 언어만 제공한다.
- 알림 설정 진입점: 제공하지 않는다.

### 콘텐츠

- 서비스 소개 영상 / 데모: 제공하지 않는다.
- FAQ / 도움말 섹션: 제공하지 않는다.
- 가격 정책 / 플랜 안내: 제공하지 않는다.
- SNS 공유 버튼: 제공하지 않는다.
- 뉴스레터 구독 / 이메일 수집 폼: 제공하지 않는다.

### 네비게이션

- 상단 GNB(Global Navigation Bar): 홈 페이지에는 별도 네비게이션 바를 두지 않는다.
- 푸터(Footer): 구현하지 않는다.
- 사이드바: 구현하지 않는다.
- 뒤로 가기 처리: 홈은 히스토리 스택의 시작점이므로 별도 처리 없음.

### 접근성 (이번 버전)

- 스크린 리더 최적화: 기본 시맨틱 마크업 수준만 제공하고, 전용 접근성 테스트는 다음 버전에서 진행한다.
- 키보드 단축키: 제공하지 않는다.

### 성능 최적화 (이번 버전)

- 이미지 Lazy Loading: 이번 버전에서는 이미지 에셋이 없으므로 해당 없음.
- 번들 최적화 / Code Splitting: 홈 페이지 단독 최적화는 하지 않으며, Vite 기본 빌드 설정을 따른다.
- PWA / 오프라인 지원: 구현하지 않는다.

## 용어 정의

| 용어                     | 정의                                                                                                               |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------ |
| **홈 페이지**            | 사용자가 서비스에 처음 진입했을 때 표시되는 루트(`/`) 경로의 화면                                                  |
| **문서 링크 카드**       | 학습 자료(공식 문서)로 이동하는 외부 링크를 카드 형태로 표시한 UI 컴포넌트                                         |
| **시작 버튼**            | 퀴즈 목록 화면(`/quiz`)으로 이동하는 CTA(Call To Action) 버튼                                                      |
| **소개 문구**            | 헤더 아래에 표시되는 서비스 한 줄 설명. 상수로 관리되며 미확정 상태                                                |
| **페이드인 애니메이션**  | 페이지 진입 시 콘텐츠가 투명도 0에서 1로 전환되며 나타나는 진입 애니메이션                                         |
| **모바일 퍼스트**        | 모바일 화면 크기를 기준으로 먼저 설계하고, 데스크탑 화면에서도 자연스럽게 동작하도록 반응형으로 확장하는 개발 전략 |
| **prefers-color-scheme** | 운영체제/브라우저의 라이트/다크 모드 설정을 감지하는 CSS 미디어 쿼리                                               |
| **CTA**                  | Call To Action. 사용자에게 특정 행동(여기서는 퀴즈 시작)을 유도하는 UI 요소                                        |
| **상수 배열**            | 코드에서 변경 없이 관리되는 고정 데이터 목록. 문서 링크 카드 목록과 소개 문구가 이에 해당                          |
| **정적 UI**              | 서버 API를 호출하지 않고 하드코딩된 데이터만으로 렌더링되는 화면                                                   |
