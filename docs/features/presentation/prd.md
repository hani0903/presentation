# 프레젠테이션 페이지 PRD

> 참조 스펙: `docs/features/presentation/spec-fixed.md`
> 작성일: 2026-05-30

---

## 개요

### 목적

홈 페이지에서 발표 카드를 클릭했을 때 이동하는 `/harness` 경로에서 슬라이드 형태의 발표 자료를 제공한다. 발표자가 브라우저에서 직접 키보드와 마우스로 슬라이드를 조작하며 발표할 수 있는 환경을 구성한다.

### 배경

현재 `/harness`는 placeholder 페이지만 존재한다. 내일 스터디 발표에서 "Claude Code Harness 기반 컴포넌트 자동화 파이프라인"을 주제로 발표하므로, 실제 슬라이드 덱으로 교체가 필요하다. 슬라이드 콘텐츠는 추후 확정 후 상수 배열 수정으로 교체 가능하도록 설계한다.

### 범위

- 대상 화면: `/harness`
- 구현 형태: 정적 슬라이드 덱 (서버 연동 없음)
- 우선 환경: 데스크톱 브라우저

### 성공 기준

- 7개 슬라이드를 키보드 방향키와 마우스 클릭으로 이동할 수 있다.
- 현재 슬라이드 번호와 전체 슬라이드 수가 표시된다.
- 슬라이드 전환 시 좌우 슬라이드 애니메이션이 동작한다.
- 첫/마지막 슬라이드 경계에서 이동이 조용히 무시된다.

## 사용자 스토리

### US-01 슬라이드 진입

> **As a** 발표자  
> **I want to** `/harness`에 진입했을 때 첫 번째 슬라이드를 보고 싶다  
> **So that** 발표를 시작할 수 있다

**수용 기준**

- 페이지 진입 시 첫 번째 슬라이드(1/7)가 16:9 비율로 화면 중앙에 표시된다.
- 슬라이드 하단에 `1 / 7` 인디케이터가 표시된다.
- 다크 모드 기반 화면이 기본으로 적용된다.

---

### US-02 키보드로 슬라이드 이동

> **As a** 발표 중인 발표자  
> **I want to** 키보드 방향키로 슬라이드를 이동하고 싶다  
> **So that** 마우스 없이 발표에 집중할 수 있다

**수용 기준**

- `ArrowRight` 입력 시 다음 슬라이드로 전환된다.
- `ArrowLeft` 입력 시 이전 슬라이드로 전환된다.
- 전환 시 좌우 슬라이드 애니메이션이 동작한다.
- `prefers-reduced-motion` 환경에서는 즉시 교체된다.
- 첫 번째 슬라이드에서 `ArrowLeft`를 누르면 아무 동작도 하지 않는다.
- 마지막 슬라이드에서 `ArrowRight`를 누르면 아무 동작도 하지 않는다.

---

### US-03 마우스로 슬라이드 이동

> **As a** 발표 중인 발표자  
> **I want to** 화살표 버튼을 클릭해 슬라이드를 이동하고 싶다  
> **So that** 키보드 없이도 발표를 진행할 수 있다

**수용 기준**

- 슬라이드 좌우 가장자리에 오버레이 방식으로 이전/다음 화살표 버튼이 표시된다.
- 호버 시 버튼이 강조된다.
- 첫 번째 슬라이드에서 이전 버튼이 비활성화된다.
- 마지막 슬라이드에서 다음 버튼이 비활성화된다.

---

### US-04 진행 상황 파악

> **As a** 발표를 보는 청중 또는 발표자  
> **I want to** 현재 몇 번째 슬라이드인지 알고 싶다  
> **So that** 발표 진행 상황을 파악할 수 있다

**수용 기준**

- 슬라이드 박스 하단 중앙에 `현재 / 전체` 형식의 인디케이터가 표시된다.
- 슬라이드 이동 시 인디케이터가 즉시 업데이트된다.

---

### US-05 콘텐츠 교체

> **As a** 발표 내용을 확정한 발표자  
> **I want to** 슬라이드 콘텐츠를 쉽게 수정하고 싶다  
> **So that** 코드 구조 변경 없이 발표 내용을 업데이트할 수 있다

**수용 기준**

- 슬라이드 데이터가 상수 배열 한 곳에서 관리된다.
- 상수 배열 수정만으로 슬라이드 수와 내용이 변경된다.

## 기술 결정 (ADR)

> 채택 구현 안: **안 C — `widgets/slide-deck` 캡슐화형**
>
> 슬라이드 덱은 렌더링·네비게이션·키보드·상태·애니메이션이 강하게 결합된 독립적인 UI 시스템으로 판단하여, `widgets/slide-deck` 하나로 캡슐화한다. `pages/harness`는 `<SlideDeck slides={SLIDES} />` 한 줄 수준으로 유지한다.

---

### ADR-01 슬라이드 상태 관리 위치

| 항목             | 내용                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Context**      | 현재 슬라이드 인덱스, 이전/다음 이동 로직, 경계 처리, 키보드 이벤트 처리를 어디서 담당할지 결정해야 한다. 슬라이드 덱의 모든 인터랙션은 하나의 상태(currentIndex)를 중심으로 연결되며, 여러 컴포넌트(SlideView, SlideControls, SlideIndicator)가 이 상태를 공유한다.                                                                                                                                                                                     |
| **Decision**     | `widgets/slide-deck/model/useSlideNavigation.ts`에 currentIndex 상태와 goNext/goPrev 액션, 키보드 이벤트 리스너를 모두 정의한다. `SlideDeck.tsx`가 이 훅을 호출하고 하위 컴포넌트에 props로 전달한다. 상태는 Widget 내부에 완전히 캡슐화되며 외부에 노출되지 않는다.                                                                                                                                                                                     |
| **Alternatives** | ① **`pages/harness`에서 직접 관리** — 거부 이유: 페이지 레이어에 네비게이션 로직이 결합되어 페이지의 책임이 커진다. 안 A의 단점과 동일. ② **`features/navigate-slide` 레이어 분리** — 거부 이유: 슬라이드 네비게이션은 발표 페이지 외에 재사용 가능성이 낮다. 레이어를 늘리는 ceremony 대비 이득이 없다. ③ **Zustand store** — 거부 이유: 서버 상태도 아니고 여러 페이지에서 공유할 필요도 없는 로컬 UI 상태에 전역 store를 사용하는 것은 과도한 설계다. |
| **Consequences** | ✅ SlideDeck에 slides prop만 주입하면 전체 네비게이션 기능을 독립 테스트 가능. 상태가 Widget 내부에 캡슐화되어 외부 의존성 없음. ⚠️ currentIndex에 외부에서 접근하거나 제어(URL 기반 딥링크, 발표자 모드 등)가 필요해지면 props로 제어권을 올리거나 Context를 추가해야 한다. 현재 요구사항에는 해당 없음.                                                                                                                                                |

### ADR-02 슬라이드 전환 애니메이션 구현 방식

| 항목             | 내용                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ---------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Context**      | 슬라이드 이동 시 좌우 슬라이드 애니메이션이 필요하다. 다음 슬라이드는 오른쪽에서, 이전 슬라이드는 왼쪽에서 밀려들어온다. `prefers-reduced-motion` 환경에서는 즉시 교체해야 한다. 런타임 JS 비용을 최소화해야 한다.                                                                                                                                                                                                                              |
| **Decision**     | CSS `transform: translateX()`와 `transition`을 이용한 순수 CSS 전환을 사용한다. 이동 방향(`direction: 'next' \| 'prev'`)을 state로 관리하고, 방향에 따라 진입·퇴장 위치(`translateX(100%)` 또는 `translateX(-100%)`)를 결정한다. `@media (prefers-reduced-motion: reduce)`에서 transition을 제거해 즉시 교체로 fallback한다.                                                                                                                    |
| **Alternatives** | ① **Framer Motion** — 거부 이유: 번들 크기(~50KB) 대비 단순 좌우 전환에 과도한 의존성. 외부 라이브러리 추가 없이 CSS로 충분히 구현 가능. ② **CSS `@keyframes` animation** — 거부 이유: 방향별로 keyframe을 별도 정의해야 하고, 슬라이드 진입/퇴장을 동시에 제어하기 어렵다. transition이 더 단순하다. ③ **JS `requestAnimationFrame`** — 거부 이유: CSS transition으로 해결되는 것을 JS로 처리하면 불필요한 복잡도와 메인 스레드 부담이 생긴다. |
| **Consequences** | ✅ 런타임 JS 없는 CSS 전환. `prefers-reduced-motion` 처리를 CSS 한 줄로 해결. 외부 라이브러리 의존성 없음. ⚠️ 슬라이드 진입·퇴장 두 요소를 동시에 화면에 올려야 하므로 overflow 처리(`overflow: hidden`)가 필요하다. 방향 state를 잘못 초기화하면 첫 진입 시 불필요한 애니메이션이 발생할 수 있어 초기값 처리를 주의해야 한다.                                                                                                                  |

### ADR-03 슬라이드 데이터 구조 및 콘텐츠 렌더링

| 항목             | 내용                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Context**      | 슬라이드 데이터(제목, 핵심 키워드)를 어디에 정의하고, 향후 HTML 파일·MDX·CMS·API로 콘텐츠 소스가 변경될 때 인터페이스를 어떻게 유지할지 결정해야 한다. `SlideDeck`은 데이터 소스에 무관하게 동작해야 한다.                                                                                                                                                                                                                                                                                                                              |
| **Decision**     | `entities/slide/model/types.ts`에 `Slide` 인터페이스를 정의하고, `entities/slide/model/constants.ts`에 `SLIDES` 배열을 작성한다. `SlideDeck`은 `slides: Slide[]` prop만 받으며 데이터 소스를 알지 못한다. `HarnessPage`가 `SLIDES` 상수를 import해 `SlideDeck`에 주입한다. 향후 소스가 변경되면 `HarnessPage`의 주입 방식만 교체하면 된다.                                                                                                                                                                                              |
| **Alternatives** | ① **`widgets/slide-deck` 내부에 상수 배열 포함** — 거부 이유: 데이터와 렌더링이 결합되어 콘텐츠 소스 변경 시 Widget 수정이 필요하다. "Widget은 slides prop만 받는다"는 설계 원칙에 위반. ② **`shared/config`에 슬라이드 배열 배치** — 거부 이유: `shared`는 도메인 무관 코드만 위치해야 한다는 FSD 원칙 위반. 슬라이드는 발표 도메인 데이터다. ③ **`Slide` 타입에 `contentHtml: string` 직접 포함** — 거부 이유: 지금은 텍스트 기반이고 HTML 연결은 향후 요구사항이다. 미사용 필드를 현재 타입에 추가하면 타입이 불필요하게 복잡해진다. |
| **Consequences** | ✅ `SlideDeck`은 데이터 소스와 완전히 분리. 상수 배열 수정만으로 슬라이드 수·내용 변경. API 전환 시 `HarnessPage`의 주입 방식만 교체하면 되며 Widget은 수정 불필요. ⚠️ 현재 Slide 타입이 `title: string`, `bullets: string[]` 수준이면 향후 HTML·MDX 콘텐츠를 담으려면 타입 확장이 필요하다. 타입 변경 시 기존 상수 배열과 SlideView 컴포넌트도 함께 수정해야 한다.                                                                                                                                                                     |

### ADR-04 키보드 이벤트 처리 위치

| 항목             | 내용                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ---------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Context**      | `ArrowLeft`/`ArrowRight` 키보드 이벤트를 어디서 처리할지 결정해야 한다. 이벤트는 `document` 레벨에서 발생하지만, 네비게이션 로직은 SlideDeck 내부 상태와 연결되어 있다. 슬라이드 덱이 마운트된 동안만 이벤트를 수신해야 하며, 언마운트 시 정리되어야 한다.                                                                                                                                                                                                                                            |
| **Decision**     | `useSlideNavigation` 훅 내부에서 `useEffect`로 `document.addEventListener('keydown', handler)`를 등록한다. cleanup 함수에서 `removeEventListener`로 해제한다. 훅이 마운트/언마운트 생명주기와 자동으로 연동되므로 이벤트 누수가 없다.                                                                                                                                                                                                                                                                 |
| **Alternatives** | ① **`HarnessPage`에서 `onKeyDown` prop으로 전달** — 거부 이유: `div`의 `onKeyDown`은 포커스가 해당 요소에 있어야 발화한다. 발표 중 포커스가 다른 요소로 이동하면 키보드 이동이 동작하지 않는다. ② **전역 Zustand store에서 이벤트 구독** — 거부 이유: 로컬 UI 인터랙션을 전역 store에 연결하는 것은 과도한 설계다. ③ **`SlideDeck.tsx`에서 직접 `useEffect` 처리** — 거부 이유: 컴포넌트에 로직이 혼재된다. `useSlideNavigation`에 격리하면 훅 단독으로 테스트할 때 키보드 동작도 함께 검증 가능하다. |
| **Consequences** | ✅ 훅 마운트/언마운트와 이벤트 리스너 생명주기가 자동으로 일치. 포커스 위치와 무관하게 키보드 동작. 훅 단위 테스트에서 `keydown` 이벤트를 dispatch해 동작 검증 가능. ⚠️ `document` 레벨 이벤트이므로, 발표 중 다른 컴포넌트가 동일한 키(ArrowLeft/Right)를 사용한다면 이벤트 충돌이 발생할 수 있다. 현재 `/harness` 페이지에는 해당 컴포넌트가 없으므로 이슈 없음.                                                                                                                                    |

## Out of Scope

### 콘텐츠

- 실제 발표 내용 완성: 콘텐츠 확정은 이번 구현 범위 밖이다. placeholder 수준으로 채운다.
- HTML 파일 외부 로딩: 슬라이드 콘텐츠를 별도 HTML 파일에서 fetch하는 기능은 구현하지 않는다.
- 이미지/영상 에셋 삽입: 슬라이드에 이미지나 동영상을 포함하지 않는다.
- 코드 하이라이팅: 슬라이드 내 코드 블록 문법 강조는 구현하지 않는다.

### 발표자 도구

- Speaker Note(발표자 노트): 구현하지 않는다.
- 발표자 모드(타이머, 다음 슬라이드 미리보기 등): 구현하지 않는다.
- PDF Export: 구현하지 않는다.
- 전체화면 API: 구현하지 않는다. 브라우저 기본 전체화면 사용.

### 네비게이션 확장

- 슬라이드 목차/썸네일 뷰: 구현하지 않는다.
- 슬라이드 번호 직접 입력으로 이동: 구현하지 않는다.
- 터치/스와이프 제스처: 모바일 최적화 미포함이므로 구현하지 않는다.
- 스페이스바/엔터키 이동: `ArrowLeft`/`ArrowRight`만 지원한다.

### 다중 발표

- 발표 목록 선택: `/harness`는 `Design to Commit` 단일 발표만 제공한다.
- 발표 간 이동: 다른 발표로 이동하는 내비게이션은 제공하지 않는다.
- 발표 공유/임베드: 구현하지 않는다.

### 서버 연동

- 슬라이드 데이터 API: 서버에서 슬라이드를 fetch하지 않는다. 상수 배열로 관리한다.
- 발표 조회수/통계: 구현하지 않는다.
- 사용자 진행 상태 저장: 마지막으로 본 슬라이드를 기억하지 않는다.

### 반응형 / 접근성

- 모바일 최적화: 데스크톱 발표 환경이 우선이므로 모바일 레이아웃은 구현하지 않는다.
- 스크린 리더 최적화: 기본 시맨틱 마크업 수준만 제공한다.
- 고대비 모드: 구현하지 않는다.

## 용어 정의

| 용어                         | 정의                                                                                          |
| ---------------------------- | --------------------------------------------------------------------------------------------- |
| **슬라이드(Slide)**          | 발표 자료의 한 장. 제목, 핵심 키워드 목록으로 구성된다.                                       |
| **슬라이드 덱(Slide Deck)**  | 여러 슬라이드를 묶은 발표 자료 전체.                                                          |
| **슬라이드 박스**            | 브라우저에서 슬라이드를 표시하는 16:9 비율의 컨테이너 영역.                                   |
| **인디케이터**               | 슬라이드 하단 중앙에 표시되는 `현재 / 전체` 형식의 진행 상황 표시 요소.                       |
| **오버레이 컨트롤**          | 슬라이드 박스 위에 반투명으로 겹쳐 있는 이전/다음 화살표 버튼.                                |
| **슬라이드 전환**            | 슬라이드가 교체될 때 발생하는 좌우 슬라이드 방향의 이동 애니메이션.                           |
| **경계 동작**                | 첫 번째 슬라이드에서 이전, 마지막 슬라이드에서 다음을 시도할 때 아무 동작도 하지 않는 처리.   |
| **placeholder 콘텐츠**       | 실제 발표 내용 확정 전까지 구조를 보여주기 위해 채우는 제목 + 핵심 키워드 수준의 임시 콘텐츠. |
| **`prefers-reduced-motion`** | 운영체제에서 애니메이션 축소를 요청하는 미디어 쿼리. 이 환경에서는 슬라이드를 즉시 교체한다.  |
