# 디자인에서 커밋까지 — AI 기반 컴포넌트 자동화 파이프라인

> 발표자: 조하은 | 발표 시간: 5분

---

## 슬라이드 1 — 문제 제기

### "디자이너가 컴포넌트 스펙을 바꾸면, 개발자는 어떻게 알 수 있을까?"

지금 팀의 현실:

1. 디자이너가 Figma에서 색상 토큰을 수정한다
2. 개발자는 모른다
3. 코드 리뷰에서 "이거 디자인 시스템 색이에요" 코멘트가 달린다
4. 수정하고 다시 PR 올린다

**문제의 본질**: 디자인-코드 싱크를 사람이 수동으로 맞추고 있다

---

## 슬라이드 2 — 제안하는 파이프라인

```
┌──────────────────────────────────────────────────────────────┐
│  Figma  ──→  design-md-generator  ──→  design.md            │
│  (디자인 작업)    (Figma 플러그인)        (토큰·규칙 문서)    │
└──────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────┐
│  Claude Code 하네스 (AI)                                     │
│                                                              │
│  design.md 읽기                                              │
│      │                                                       │
│      ├──→  공통 컴포넌트 생성  (토큰 자동 적용)              │
│      │         │                                             │
│      │         └──→  PostToolUse 훅 (저장 즉시 위반 감지)   │
│      │                                                       │
│      └──→  Storybook story 자동 생성                         │
│                (컴포넌트 props 기반 args·controls 작성)      │
└──────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────┐
│  Husky (커밋 게이트)                                         │
│                                                              │
│  git commit 시도                                             │
│      │                                                       │
│      ├──→  pre-commit                                        │
│      │         ├── ESLint + TypeScript 검사                 │
│      │         └── Storybook 빌드 가능 여부 확인            │
│      │                                                       │
│      └──→  commit-msg                                        │
│                └── Conventional Commits 형식 검증           │
│                    feat / fix / chore: 메시지               │
│                                                              │
│  ✅ 통과 → 커밋 완료        ❌ 실패 → 수정 요청             │
└──────────────────────────────────────────────────────────────┘
```

## 슬라이드 3 — 각 기술이 뭔지

### Figma design-md-generator

- Figma 플러그인
- 팀의 디자인 토큰(색상, 타이포, 간격), 컴포넌트 규칙을 `.md` 파일로 추출
- 사람이 손으로 정리하던 디자인 문서를 자동화

### Claude Code 하네스

- AI 코드 에디터인 Claude Code의 자동화 시스템
- `design.md`를 읽고, 컴포넌트 생성 시 디자인 토큰을 자동으로 적용
- 파일 저장할 때마다 훅이 실행되어 위반 패턴(하드코딩된 색상, 임의 픽셀 등)을 즉시 감지

### Storybook

- 컴포넌트 문서화 도구
- AI가 컴포넌트 props를 분석해서 story 파일까지 자동 생성
- 사람이 직접 `args`와 `controls`를 쓰는 비용 제거

### Husky

- Git 훅 관리 도구
- `pre-commit`: 커밋 전 린트·타입 체크 자동 실행, 실패하면 커밋 차단
- `commit-msg`: 커밋 메시지가 `feat: 로그인 구현` 형식인지 검증
- 팀 전체가 규칙을 지키도록 강제하되, 사람이 신경 쓸 필요가 없음

## 슬라이드 4 — 지금 실제로 돌아가는 것

현재 loopin 프로젝트에서 이미 구현된 부분:

**Claude Code 하네스 + Husky**

```
.tsx 파일 수정
    │
    ▼
PostToolUse 훅 자동 실행
    │
    ├── #3B82F6 같은 hex 하드코딩 감지 → Claude가 즉시 수정
    ├── shadow-xl 감지 → 경고
    └── bounce 이징 감지 → 경고

git commit 시도
    │
    ├── pre-commit: ESLint + TypeScript 통과해야 커밋 가능
    └── commit-msg: "feat: ~" 형식 아니면 커밋 거부
```

팀원이 디자인 시스템 규칙을 외우지 않아도, 시스템이 알아서 잡아준다.

## 슬라이드 5 — 도전해보고 싶은 것

아직 연결하지 못한 두 가지:

**① Figma → design.md 자동 추출**

- 현재는 design.md를 사람이 직접 작성
- design-md-generator 플러그인을 연결하면 Figma 저장 시 자동으로 갱신
- 디자이너가 Figma를 수정하는 순간, 개발 규칙도 같이 업데이트됨

**② Storybook story 자동 생성**

- 현재는 컴포넌트 작성 후 story를 사람이 따로 작성
- AI가 컴포넌트 props를 읽고 story 초안을 자동 생성
- Storybook을 팀이 실제로 쓰게 만드는 진입 장벽을 낮춤

## 슬라이드 6 — 왜 이게 중요한가

| 작업                  | 지금                             | 파이프라인 도입 후       |
| --------------------- | -------------------------------- | ------------------------ |
| 디자인 토큰 코드 반영 | 눈으로 Figma 확인 후 수동 작성   | design.md 기반 자동 생성 |
| 디자인 규칙 위반 발견 | 코드 리뷰 때 발견                | 저장 즉시 감지           |
| Storybook 문서 작성   | 컴포넌트 완성 후 별도 작성       | story 자동 생성          |
| 커밋 메시지 형식 통일 | 팀 규칙 공유 후 각자 지키길 기대 | Husky가 강제             |

**핵심**: 사람이 규칙을 기억하고 지키는 구조 → 시스템이 강제하는 구조
